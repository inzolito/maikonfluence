11196 OP PRO 4.x Code Standardization & Feature Parity : Mangoola | Identification Phase

Created by GEORGE Nathan, last modified on jul 17, 2024

Initial identification of customizations and feature gaps of Shell App panels commenced as part of Project Leap (OAS + OP Pro Integration).

Mangoola was a potential field trial candidate for Leap PR1 however due to the level customization and upgrade effort it was decided to abandon the Leap PR1 approach and move to PR2.

Refer to Glencore Mangoola section below.


Overview

This page aim to give high-level plan on what and how the Leap Implementation (OP Pro and OAS Integration) will be executed on customer site.

Installation Strategy

The main intention for the installation is an integrated system between OP Pro and OAS, but the solution released should also compatible to run each product as a standalone system.

On situation where customer already has both system running:

  • To mitigate any risk during the integration, the execution plan separated into stages and validated before moving to next steps. Primarily, this is to ensure existing equipment on both system could still communicate and no regression introduced.
  • The integration process started from server, and continues with equipment only after server validation completed.

Jigsaw Web Service (JWS)

Mangoola has JWS installed on both servers, listening on port 8888 and configured to use protosql on port 1111 on the active cluster IP address (10.168.30.250). It will be used by the MTI third-party interface, which should be connecting to the backup cluster address (to reduce processing load on the active server).

JWS is installed in /opt/local/JigsawWebService and should thus be unaffected by any future Jigsaw updates. JWS versions are not currently explicitly defined, but the JWS build installed was taken from http://10.76.9.64:8081/artifactory/list/jigsaw-webservice/latest/master/ on 20190604. JWS is configured per the standard instructions to start automatically via supervisord; start-up after reboot has not yet been tested on the servers, but was tested on a simulator following the same install process.

17 oct 2019 JWS confirmed to start automatically on secondary server after reboot, although didn't succeed on primary. Killed the process (as supervisorctl stop didn't kill it), then restarted.

Reverse Tunnel

Hexagon Mining connection to Mangoola Jigsaw servers is based on reverse SSH tunnel:

  • /etc/init.d/SSHTunnel running in Mangoola secondary server (and checked every 10 minutes), which will create remote port forward on port 2271 to Mangoola server
  • /etc/init.d/SSHTunnel running in Mangoola primary server (and checked every 10 minutes), which will create remote port forward on port 2270 to Mangoola server
  • SSHTunnel connects to DMZ server running with IP 10.76.9.163 (DMZ login) and 27.127.195.38 port 20202 (User: ssh_guest, No password from Mangoola server)
  • Then cctgw creates the above port forwards using ssh connection from DMZ server

So if port 2270 or 2271 are not in the listening port list (netstat -npl) in DMZ server, this means SSHTunnel script is not running in Mangoola secondary server or the primary server. To start it, execute the following in Mangoola secondary server:

sudo /etc/init.d/SSHTunnel stop
sudo /etc/init.d/SSHTunnel start

Once it is up and running, start the ssh tunnel from cctgw to DMZ server.
