GVOS Database Locks Issue

IN-5858 - Autenticar para ver datos del tique

9 feb 2024 Usuario desconocido (ojo) 

The GVOS errors were mostly "database is locked" issues.

This error occurs because SQLite uses Posix file locking to share the database between multiple processes. In all OAS and OpPro installations, there are only two processes that directly access the SQLite database file:

  • gv_main - the main GVOS application server process

  • gvcleaner - a background process which saves a clean copy of the sqlite database every 5 minutes.

Each query submitted by gv_main to the database waits for up to 5 seconds when SQLite returns a BUSY error indicating that another process has the database locked.  If the database is locked for more than 5 seconds, the query returns with a the 'database is locked' error. 

I have found on OAS deployments that the Micron 32GB eMMC flash used on all Core computers has severe performance issues whenever the on-chip FTL performs block garbage collection.   In many cases, the eMMC is locked for over 10 seconds causing timeout errors in GVOS.  We discovered the same performance issues using the 16GB Micron flash chips on OASLV, so we negotiated with Variscite to only supply Samsung eMMC chips that do not exhibit this performance slowdown.  These flash slowdowns are random and caused by the Micron FTL which has  insufficient buffering on 32G and smaller flash chips. 

On OAS OpenWRT builds, we use the f2fs file system which performs much better than the standard ext4 file system used on the Yocto 32 release.  The better performance of f2fs is because eMMC chips support a TRIM command which allows the file system to inform the eMMC memory chip of blocks which are no longer used.  The ext4 file system was designed for rotational disks and SSDs and performs poorly on eMMC.  Supporting TRIM in f2fs allows the FTL to pre-erase unused blocks eliminating most of the garbage collection delays.  Using an ext4 file system with the default "data=ordered" configuration exacerbates the performance delays of eMMC chips.

To speed access to the on-board eMMC, we can try one of the following options.

  1. If using ext4, change the mount options to "data=writeback".

  2. Reformat the data partition on which the SQLite database is storesd as f2fs.  On Yocto 32, this will probably require adding f2fs to the Linux kernel.

  3. Store the active SQLite database in RAM, allowing gvcleaner to write a non-volatile copy to eMMC.  Doing this would speed database access but lose up to 5 minutes of data after a reboot.

  4. Shrink the size of the on-board database.  Currently, the largest database table is "phrases" which contains allot of MineOPS phrase translations that are never required on-board.  Setting the 'device' for each phrase and allowing phrases to replicate only those phrases necessary for on-board use will significantly decrease the OpPro on-board database size and decrease the delay for database synchronization.
