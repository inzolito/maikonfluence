Automatic Haul Cycle Configurations

Automatic Haul Cycle Configurations

See Auto Haul Matrix for specific sensor values

Intro
The purpose of this page is to outline how OEM interfaces are used to automatically change the activity of trucks during their haul cycle.
It has been noted that regional offices use different methods to do this that may not be optimal as the standard setups have been designed to be used out of the box.


Overview
OP Pro uses sensor types to designate sensors that will be used for integration purposes, whether that is a functional action or simply a data store.
In case of Asset Health being integrated with OP Pro, there are base sensor types also called "known sensors" that share information to OP Pro by the same rules above.
In OP Pro there is capability to use only GPS for all activity changes. This will work; however, it is not as accurate as the OEM interface. In OP Pro both methods can be enabled. It will use the OEM interface first and if that does not trigger the activity it will use GPS.


There is also a configuration to set GPS distance to 0 for activities in order to force activity changes from the OEM sensor by itself. This however does not seem to be used as the redundancy of the two options together makes for a smoother transition. If the interface needs maintenance and is acting up the machine should still reliably cycle by GPS.


Note on gear mapping
It is worth noting that there is a configuration in OP Pro for VIMS called "dynamic_gear_numbers" which can be used to override the default values map for gear_position sensor for the interface.
A values map is the raw value we receive from the interface vs the actual gear value. A default values map is already built into the interface configuration. If issues are seen with it, you can have an operator cycle gears in a specified order, read the sensor value, and correct the mapping or have site contact the OEM dealer for an updated values map for the machines running and add the configuration to the interface device in OP Pro.

Activity Changes
From the User Reference Manual, we see the following matrix:


The primary Activities are shown above:
Traveling > Waiting at Shovel > Spotting > Loading > Hauling > Queuing at Dump > Backing > Tipping
Several of the Activities can be triggered by OEM interfaces or other input, outlined below the sensor type being used for activity transitions, by each interface.


Caterpillar Trucks - VIMS Std | VIMS3G

  • payload_status
    • Traveling, Loading, Hauling, Tipping
  • gear_position
    • Spotting, Backing
  • GPS
    • Waiting at Shovel, Queuing at Dump

The VIMS interface uses a sensor called Payload Status/OHT Payload Status which offers different values for the haul state. We are using four of the payload state values to change the activity onboard. The gear sensor is also used for two activity changes of the main activity changes. Finally, the last two activities use GPS to determine when to change the activity.


Komatsu Trucks - (PLM2/3 | PLMIV) + VHMS (Komtrax Plus)

  • payload_status
    • (Traveling, Loading, Hauling, Tipping)(PLMIV) |  (Loading, Tipping) (PLM2/3)
  • gear_position (VHMS - not implemented for activity changes)
    • Spotting, Backing
  • Tray Down (PLM2/3)
    • Traveling
  • Final Payload (PLM2/3)
    • Hauling
  •  GPS
    • Waiting at Shovel, Queuing at Dump

The PLMIV interface we are able to utilize Payload Status for four of the main activities. There isn't access to the gear position sensor in PLM, but it's counterpart VHMS (Komtrax Plus) includes it for potential use in Spotting and Backing activities, code has not been written to handle this.  In OP Pro PLM2/3 we use vehicle_status instead of payload_status in the code for the sensor type.  PLM2/3 use different sensors for Traveling and Hauling as shown above.


Hitachi Trucks - Hitachi5 | Hitachi7 (Note:  In recent tests of Hitachi trucks payload_status data received is not consistent and is removed in default config sheets for AH)

  • payload_status
    • Traveling, Loading, Hauling, Tipping
  • gear_position
    • Spotting, Backing
  • GPS
    • Waiting at Shovel, Queuing at Dump

Hitachi interfaces have payload_status and gear_position sensors as well. There was some work done recently for Hitachi5 for HTM trucks that needs to be updated. Hitachi7 there is a known issue with gear sensor being sent as a string when deployed in Asset Health.


Liebherr Trucks - LiebherrCan/LiebherrPayload

  • LiebherrPayload sends events that trigger auto activity changes (Not confirmed to be used/working yet)
    • Traveling, Loading, Hauling, Tipping (does not use sensor_type/known_sensor)
    • GPS for Spotting, Backing, Waiting at Shovel, Queuing at Dump


Belaz Trucks - Belaz

  • gear_position
    • Spotting, Backing
  • dipper
    • Loading
  • body_lever
    • Tipping
  • GPS
    • Traveling, Waiting, Hauling, Queued

Belaz interface does not provide a payload sensor to trigger hauling. Once hauling is detected through GPS, the last dipper value is set as payload.

Auto Haul Matrix

Note GPS activity distances listed are a starting point, but may differ from site to site for optimal setting.  LiebherrPayload interface not verified to be used as the code suggests.  VHMS has gear_position functionality not implemented currently.

For gear_position sensor type, there can be different raw integer values that correspond to R (reverse) in a values map.  Gear position values used to activity changes are hardcoded in JInterface files for associated interfaces.


Note that when using JHealthIntegration conversions are made for payload_status to general values:

JHealthIntegration payload_status sensorValue
PayloadStatus_Traveling 1
PayloadStatus_Loading2
PayloadStatus_Hauling3
PayloadStatus_Tipping4

Example:  VIMS interface sends payload_status = 6 for Traveling activity, this would show as payload_status=1 for units loaded with Asset Health.  Functionally the business logic is the same.
