HxGN MineOperate : How to Slice a User story

Created by GARCIA Janssen on abr 05, 2024

When we write user stories, they can be broad and that is Ok! These user stories should be broken down into smaller and workable chunks by the team when it makes sense.

Splitting user stories is an important part of the work done by an agile team. Leaving a user story too big makes it difficult for it to be completed faster. Working on smaller user stories helps teams to get a faster feedback loop, as they can be completed faster, the product owner, stakeholders and customers can test and give feedback as soon as possible, which will help the team easily improve and correct things quicker.

How can we slice a user story?

Horizontal Vs Vertical Slicing

Rather than giving a customer or an end-user one layer of a cake each time (as seen in the Horizontal slicing above i.e., User interface layer done in Sp 1) in which case the client only gets to taste one particular layer at a time, In agile, we want to give the end-user a tiny piece of the whole cake hence the name Vertical Slicing (As seen in the picture above - SP 1- Subject).

In a horizontal slicing, the end-user will only get the chance to experience the system only when it is fully done. For example, we create the first the Database Layer, then the security layer, then the middle tier and finally the UI layer however in a vertical slicing, the idea is to build the system user story by user story where each story will cross all horizontal layers.

With the horizontal slicing, the feedback loop is long which significantly increases the risk as the entire cake must be fully baked before the customer realizes that the cake, he got is not necessarily the one he wanted. Maybe the customer preferred vanilla flavor rather than chocolate in one of the layers, if he had the chance to taste one slice, he would have realized those gaps earlier on.


SPIDR method of slicing


The SPIDR method is one method we as a team can use to split our user stories.


  • SPIKES- This is a term used in agile software development. Spikes are used to prototype the implementation of a functionality and is typically used for the evaluation and feasibility of new technologies. When teams need to build the functionality using either an old technology or a new technology, we can create spikes. Spikes involve investigations and build in knowledge. 

Other techniques should first be used to split user stories into smaller sizes before thinking about a Spike. Spike should be used to split a user story only if the team didn't manage to use other methods to split the user story. Once there is better understanding of the user story and the technology to be used then we can work on splitting the initial user story using the other methods of SPIDR.

  • PATHS- Consider the different paths. If there are several possible alternative paths in a user story, one option is to create separate user stories from some of these paths. Note: It is not absolutely necessary to write a user story for each individual path but just where it makes sense for example

             Big User story: As a customer, I want to be able to pay for my goods online so that I do not have to transfer money.

This large story can be split by using the happy flow or Unhappy flow. The above large story can be broken down using these paths:

 Happy Flow- The happy flow describes how functionality behave when everything goes well.

             Small story: As a customer, I want to be able to have a validation screen so that I can have full information on my order.

 Unhappy Flow- If there are any deviations, exceptions or other problems, the 2nd user story could then be.

             Small story: As a customer, I want to be able to see an error screen so that I can have a proof if my account is debited. 

We don't have to implement the full system but only the part that shows that something went wrong. The above stories can still be further broken down in different other paths such as payment methods i.e.: PayPal or Credit Card. Credit card path can also even be broken down further i,e. User story focusing on Visa or Mastercard.

Let's break down "As a customer, I want to be able to have a validation screen so that I can have full information on my order" further using the payment methods path:

    Smaller story 1: As a customer using Paypal, I want to be able to have a validation screen so that I can have full information on my order.

    Smaller story 2: As a customer using credit card, I want to be able to have a validation screen so that I can have full information on my order.

The above smaller stories can then be prioritized such that the "Must have" are ranked first and brought into the sprint to be Demo'd as an increment once the sprint ends i.e., Smaller US 1 can be brought in SP 1 and Smaller US 2 may not be worked on until US 3 depends on the priority as ranked by the PO.

Again, keep in mind that we should only split user stories only when it makes sense and when it is too big to fit in a sprint for example.


  • INTERFACE- Most interface have to support various platforms like mobile phones, desktops and web applications. It may be beneficial to break down large items depending on the interface. For example:

   Mobile Application: We can create user stories that relates exclusively to Android devices and then another user story on IOS devices. 

   Desktop Application: Stories can be split depending on the operating system. We can start with Windows 11, then Mac OS and then Linux

   Web Application: We can split by which browser is supported first i.e., User story on Google Chrome, then Mozilla Firefox etc.

The user stories can be prioritized depending on the target group. We can start with User stories that will affect the maximum number of users before moving to the next ones.


  • DATA- Some user stories can be split based on the datatypes they return, or the parameter they are supposed to be handled. Let's use the example of a creating a web page on banking platform where the user needs to fill some information such as 

Name - Should have text field.

Surname - Should have text field.

Date of Birth- Should have a calendar.

City- Should have a list of cities to choose from

Maximum amount of money - Should have a cursor to allow amount adjustment.

We can create 4 user stories using the data types as follows:

  User story 1: Could be to create all as text fields such that the customer can still input all the information required as a text at first.

  User story 2:  We can add the possibility for users to be able to pick dates from a calendar.

  User story 3: We can add the possibility for users to pick their choice of cities from a prepopulated list.

  User story 4: We can create a cursor for the amounts needed.

Again, do not split a user story for the sake of splitting it but only when it makes sense and when it is needed. If all 4 can be done in 1 single story and can be small enough to be completed in 1 sprint, then there would be no need to split the user story.


  • RULES- User stories usually involve a number of explicit and implicit business rules. By studying these business rules, you might discover it is possible to break an item into smaller pieces of functionality. For example, if the user story is about setting a password: "As a customer, i want to be able to set up my password so i can login securely".

We could have the following rules for password creation:

  1. Length 7 characters minimum
  2. Minimum 2 special characters
  3. Not the last 3 passwords
  4. No use of the user's name

With the above rules, it is fine if we decide not to implement the last 2 rules initially. We can allow the user to create a password with applying only the 1st two rules which can become our first user story and then the last 2 rules can be created later with another user story or even 1 user story per rule. The initial user story will be delivered in several smaller steps.

Splitting a user story by rule should be done only when it makes sense and when it is needed to make a user story smaller. 


Acceptance Criteria method of slicing a user story.

The "Splitting by Acceptance Criteria" approach involves dividing user stories based on the specific conditions that must be met for the story to be considered complete or "done." Each acceptance criterion represents a testable requirement that must be satisfied before the user story can be delivered to stakeholders.

For example, consider a user story: "As a customer, I want to add products to my cart." The acceptance criteria for this user story might include:

  • The product should be added to the cart when the "Add to Cart" button is clicked.
  • The cart icon should display the correct number of items after adding products.
  • The cart should be accessible from any page on the website.

To split this user story by acceptance criteria, we can create separate user stories for each criterion, such as:

  • “As a customer, I want the product to be added to the cart when I click the 'Add to Cart' button."
  • "As a customer, I want the cart icon to display the correct number of items after adding products."
  • "As a customer, I want the cart to be accessible from any page on the website."
